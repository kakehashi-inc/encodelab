# システム仕様 — EncodeLab

最終更新: 2026-04-30

本書は EncodeLab の機能仕様と内部設計をまとめた開発者向け資料である。要件は [要件定義.md](要件定義.md) を参照。

## 1. アプリ概要

EncodeLab は、文字列・バイナリデータの各種変換を 2 ペイン構成で双方向に行う Electron 製デスクトップツール。
中央コントロールの方向矢印で入出力を切り替え、変換ボタン押下で一度に変換を実行する (リアルタイム変換は行わない)。

## 2. 画面構成

3 カラムレイアウト。

| カラム | 役割 |
| --- | --- |
| 左ペイン | カテゴリ・タイプ選択 + データ表示領域 |
| 中央コントロール | 方向矢印 (→/←) + 変換ボタン + 変換不可理由 |
| 右ペイン | カテゴリ・タイプ選択 + データ表示領域 |

矢印の向きにより入出力の役割が決まる。

- `→`: 左ペイン = 入力 / 右ペイン = 出力
- `←`: 右ペイン = 入力 / 左ペイン = 出力

## 3. データ表示領域の種別

タイプの `display` 属性で UI が切り替わる。

| display | 入力側 UI | 出力側 UI |
| --- | --- | --- |
| `text` | 編集可能なテキストエリア + コピーボタン | 編集可能なテキストエリア + コピーボタン |
| `binary` | ファイル選択ボタン + メタ情報 + (Web 表示可能なら画像プレビュー) | バイト数 + MIME 選択 + 保存ボタン + (Web 表示可能なら画像プレビュー) |
| `image` | ファイル選択 + クリップボード貼付 + 画像プレビュー | 画像プレビュー + 保存ボタン + (QR コードの場合は生成オプション) |

## 5. 内部正規形 (CanonicalValue)

各タイプは入出力時に内部表現と相互変換する。`src/shared/conversion/domain.ts` を参照。

```ts
type CanonicalValue =
    | { kind: 'bytes'; bytes: Uint8Array }
    | { kind: 'text'; text: string }
    | { kind: 'structured'; value: unknown }
    | { kind: 'integer'; value: bigint }
    | { kind: 'image'; mime: string; bytes: Uint8Array };
```

## 6. 変換アルゴリズム

`src/renderer/conversion/engine.ts` の `runConversion` が変換の本体。

1. `parsePane(inputType, paneValue)` で入力ペインの値を `CanonicalValue` に変換。
2. `adapt(canonical, outputDomain)` で出力タイプが要求するドメインに正規化。
3. `serializePane(outputType, adapted, ctx)` で出力ペイン値を生成。

ドメイン間ブリッジは `src/renderer/conversion/adapt.ts` に集約。

- `bytes ↔ text`: UTF-8 コーデック
- `text ↔ structured`: JSON.parse / JSON.stringify
- `bytes ↔ structured`: UTF-8 → JSON
- `integer ↔ text`: `BigInt` / `toString`
- `image`: バイト列 + MIME。テキストへの adapt は UTF-8 デコード

### 6.1 ハッシュ出力の特例

ハッシュ系出力は canonical を経由せず、入力ペインの「ユーザーが見ているそのままの値」を直接ハッシュする。
これは canonical 経由だと構造化データ (JSON / YAML) が再シリアライズされ、整形空白の有無で
ハッシュ値が変わる予期せぬ挙動になるため (`src/renderer/conversion/engine.ts` の `inputBytesForHash` を参照)。

- text 表示の入力: `TextEncoder` で UTF-8 エンコードしてハッシュ
- binary 表示の入力: ファイル内容のバイト列をそのままハッシュ
- image 表示の入力: 画像バイト列をそのままハッシュ (QR では画像ファイル自体)

## 7. 変換可否判定

`src/shared/conversion/compatibility.ts` の `checkCompatibility(inputType, outputType)` が判定する。

- 入力側がハッシュ系の場合: 変換不能 (理由 `reason.hashOutputOnly`)
- それ以外: 全て変換可能

ドメイン間ブリッジが定義されているため、エンジンとしては論理的にあらゆる組合せを試行できる。
意味的に成立しない組合せ (例: ランダムなバイナリ → 整数) は実行時にエラーとして返り、出力ペインの代わりに中央エリアへエラーが表示される。

## 8. データクリア仕様

| 操作 | 両ペインのデータ |
| --- | --- |
| カテゴリ変更 | クリアする |
| タイプ変更 | クリアする |
| 方向反転 | 保持する (要件 §7) |

実装は `src/renderer/store/converter-store.ts` の `setCategory` / `setType` / `flipDirection` を参照。

## 9. 画像表示領域

`src/renderer/components/converter/ImagePreview.tsx`。

- PNG / JPEG / GIF / WebP / SVG のみプレビュー対応 (それ以外は非対応メッセージ)
- `<img>` で表示。`object-fit: contain` により縦横比を維持して領域内に収める
- 画像バイトは `Blob` 経由で `URL.createObjectURL` を使い、`useEffect` の cleanup で解放

## 10. QR コード

### 10.1 生成 (出力側 = `image / qrCode`)

ライブラリ: `qrcode` (npm)
オプション: 出力形式 (SVG / PNG)、誤り訂正レベル (L/M/Q/H)、セルサイズ (PNG 時のみ)、マージン、前景色 / 背景色。
`src/renderer/conversion/qr/generator.ts` を参照。

### 10.2 読取 (入力側 = `image / qrCode`)

ライブラリ: `jsqr`
入力経路: ファイル選択 + クリップボードからの画像貼付 (カメラは対象外)。
画像を Canvas でラスタライズして `ImageData` を取り出し、`jsQR` に渡す。
`src/renderer/conversion/qr/reader.ts` を参照。

## 11. MIME 自動判定

ライブラリ: `file-type` (ESM 専用)

- `src/main/services/mime.ts` で動的 import (`Function('return import("file-type")')`) して読み込む
- マジックバイトから MIME と拡張子を判定。判定不能時は `{}` を返し、UI 側で `application/octet-stream` (RAW) にフォールバック
- ユーザーは MIME ドロップダウンから手動上書き可能
- **data URL 出力**でも同じ MIME 判定を行うため、PNG → data URL の変換結果は `data:image/png;base64,…` の形式となる
- バイナリ出力で MIME が RAW のままでも、内容が UTF-8 として整合し制御文字が少なければ
  「テキストっぽい」と判定し、出力ペインに読取専用テキストエリアを表示する
  (`src/renderer/conversion/text-detect.ts` を参照)

## 12. ファイル入出力

`src/main/services/file.ts`

- `dialog.showOpenDialog` / `dialog.showSaveDialog` をラップ
- 読み込み結果はバイト列 (Uint8Array) で renderer に転送
- 保存はバイト列をそのまま `fs.writeFile` で書き込み

## 13. ハッシュ計算

`src/main/services/hash.ts`

- Node.js 標準の `crypto.createHash` を利用
- 対応アルゴリズム: MD5 / SHA-1 / SHA-256 / SHA-512
- 出力エンコーディング: hex / base64

## 14. IPC 一覧

`src/shared/constants.ts` の `IPC_CHANNELS` にチャンネル名を集約。`src/preload/index.ts` で `window.encodelab` として contextBridge する。

| チャンネル | 用途 |
| --- | --- |
| `app:getInfo` | アプリ情報取得 (バージョン / OS / 保存済みテーマ・言語 + OS 既定のテーマ・言語) |
| `app:setTheme` / `app:setLanguage` | 設定変更 (`userData/settings.json` に永続化) |
| `window:*` | ウィンドウ制御 (最小化/最大化/閉じる) |
| `updater:*` | 自動アップデート (electron-updater) |
| `file:open` / `file:save` | ファイル選択ダイアログ |
| `mime:detect` | MIME 自動判定 |
| `hash:compute` | ハッシュ計算 |

## 15. ライブラリ方針

| 用途 | 採用 |
| --- | --- |
| MIME 自動判定 | `file-type` (ESM, 動的 import) |
| QR コード生成 | `qrcode` |
| QR コード読取 | `jsqr` |
| ハッシュ | Node.js `crypto` |
| Base 系 / エスケープ系 / JSON↔YAML | 自前実装 (`src/renderer/conversion/`) |

メンテナンスが停止しているライブラリは採用しない方針。

## 16. 大容量バイナリへの対応

- ファイル読込・保存・MIME 判定・ハッシュ計算はメインプロセス側で実施 (renderer はバイト列のみ受け取る)
- IPC は `ipcRenderer.invoke` 経由で `Uint8Array` を直接転送 (Electron は構造化複製をサポート)
- レンダラー側のテキスト変換は同期処理だが、現実的なバイナリサイズ (数十 MB) であれば問題ない想定

## 17. 設定 / 永続化

変換履歴は保持しない (要件 §11)。

ユーザー設定 (テーマ / 言語) は `userData/settings.json` に永続化する。

| キー | 型 | 既定値 | 説明 |
| --- | --- | --- | --- |
| `theme` | `'light' \| 'dark' \| 'system'` | (未設定) | キャプションバーから手動切替。未設定なら OS のテーマに追従 |
| `language` | `'ja' \| 'en' \| 'system'` | (未設定) | キャプションバーから手動切替。未設定なら OS の言語に追従 |

実装は `src/main/services/settings.ts` (読み書き) と `src/main/index.ts` の `app:getInfo` /
`app:setTheme` / `app:setLanguage` ハンドラ。`app:getInfo` は保存値と OS 値の両方を返し、
レンダラー側は `'system'` 選択時に OS 値へ解決して UI に適用する。

## 19. ビルド・配布

### 19.1 Windows portable 版の zip 化

未署名の `.exe` を直接配布するとブラウザやウイルス対策ソフトの警告が厳しくなるため、
Windows の portable ターゲットはビルド完了後に zip に圧縮した形で配布する。

実装は `scripts/zip-portable.js` (electron-builder の `afterAllArtifactBuild` フック) と
`electron-builder.yml` の `afterAllArtifactBuild: scripts/zip-portable.js` 設定。

- フックは `BuildResult.platformToTargets` に Windows × portable ターゲットが含まれる場合のみ動作する
  (macOS / Linux ビルドでは何もせず通過する)
- `BuildResult.artifactPaths` から `.exe` を抽出し、同 basename の `.blockmap` が存在しない `.exe` のみを portable と判定する
  (NSIS は `.exe` と `.exe.blockmap` のペアを必ず出力するため、`.blockmap` が無い `.exe` は確実に portable 版)
- 該当 `.exe` を `archiver` で同名 `.zip` に圧縮し、元の `.exe` を削除する
- 生成した `.zip` のパスを返り値として返すことで、`publish` 対象に追加する
- NSIS 版の `.exe` / `.blockmap` / `latest.yml` 等、portable 以外の Windows アーティファクトは一切変更・削除しない

### 19.2 portable 実行時の自動更新スキップ

portable 版で `electron-updater` が動作すると、`latest.yml` に従って NSIS 版インストーラを
ダウンロード・実行してしまい、ユーザーが意図しない場所に通常版がインストールされる。
これを防ぐため、portable 実行時は自動更新関連処理を一切スキップする。

実装は `src/main/services/updater.ts`。

- portable 判定は `process.env.PORTABLE_EXECUTABLE_FILE` の存在で行う
  (electron-builder が portable.exe 起動時に自動設定する環境変数)
- 判定ロジックは `shouldSkipUpdater` に集約し、`initializeUpdater` / `checkForUpdates` /
  `downloadUpdate` / `quitAndInstall` / `scheduleStartupCheck` の各エントリポイントで
  早期 return する
- 既存の `isDev` ガードと同じ位置で動作するため、NSIS 版と開発実行時の挙動は変わらない
